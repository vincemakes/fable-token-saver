# fable-token-saver

[English](README.md) | **简体中文**

**Claude Code 的分层模型编排。** 让你最贵的模型(Fable、Opus——当前阵容里最强的那个)只坐镇指挥席——拆解任务、写规格、审查结果;实现工作全部交给便宜档模型(Sonnet/Haiku)。客观闸门(类型检查/测试)先把机器能抓的错全部过滤掉,贵模型一个 token 都不花在这些上面。

**版本无关**的设计:skill 只按**模型别名/档位**(最强档 / 中档 / 最便宜档)路由,从不硬编码具体 model ID,模型换代后照常工作。

## 原理

在 Claude Code 里,你选的模型只驱动*主循环*——但主循环派出的每个子代理都可以跑在另一个更便宜的模型上。模型档位之间的质量差距几乎全部在**判断力**(建什么、怎么设计结构、这个 diff 对不对),而不在把写清楚的规格敲成代码。所以:

- **Orchestrator**(你的主循环模型)写任务包、审 diff。
- **Implementer**(Sonnet 档)写代码,必须过类型检查/测试闸门才许返回。
- **Mechanic**(Haiku 档)做改名、批量编辑、配置修改——更便宜。
- **Scout**(Haiku 档)读代码库,保持 orchestrator 上下文干净。

## 两档模式——选主循环模型就是选档

不需要配置文件:skill 运行时探测主循环是什么模型,自动适配。档位名描述的是**省 token 的力度**(lite 省得少、max 省得满)——不是模型强度,类似汽车的 eco 模式。

| | **lite 档** | **max 档** |
|---|---|---|
| 主循环(`/model` 切换) | 最强档(Fable) | 中档(**推荐 Opus**) |
| 代码谁写 | Sonnet / Haiku worker | Sonnet / Haiku worker |
| 最强档干什么 | 全程掌舵每一步 | 降为**顾问子代理**,只在 2 个检查点出场:开工前设计裁决 + 合并前终审 |
| 最强档节省 | **−42%(实测)** | **Opus 主循环 −89%(实测)**;Sonnet 主循环只有 −61%,见下文 |
| 总费用(美元) | 编排方案里最低 | **比基线还贵 +86%**——本档是拿美元换最强档额度 |
| 什么时候用 | 想要顶级判断力盯每一步 | 最强档额度枯竭、且额度就是你的硬约束 |

max 档为什么成立:主循环要按主循环价为每个工具返回、每次转身簿记付钱——benchmark 里这笔"主循环税"远大于真正的判断内容(实花 17.9k vs 真决策 ~5k)。max 档把税降一档去交,最强档的判断力只买在真正拉开差距的两个时刻。最强档额度彻底枯竭时,说一句 "no fable" 可完全跳过顾问。

## 安装

```bash
git clone https://github.com/<you>/fable-token-saver ~/.claude/skills/fable-token-saver
```

Agent 工具支持按子代理覆写模型的环境(当前 Claude Code 支持)装完即用。可选:安装固定路由的 worker agent 定义:

```bash
mkdir -p .claude/agents
cp ~/.claude/skills/fable-token-saver/assets/agents/*.md .claude/agents/
```

## 使用

三种启动方式,任何档位通用:

1. **显式调用** —— `/fable-token-saver 把这个模块重构了`
2. **自然语言** —— "用 token saver 走一下"、"省token模式做这个"、"把这个委托给便宜模型做,你负责审查"
3. **主动触发** —— 大任务(约 300+ 行或 6+ 文件)时 skill 自己触发;低于委托门槛时它会刻意不掺和(见 Benchmarks——这个门槛很重要)。

## 核心机制

1. **分类** —— 机械性任务 → 最便宜档;可规格化任务 → 中档;设计密集型任务 → orchestrator 亲自做。小体量任务一律不委托。
2. **任务包** —— worker 只拿到 GOAL / CONTEXT(≤5行)/ ALLOWED FILES / SPEC / GATE / DO-NOT 围栏,绝不转发对话历史。
3. **客观闸门** —— worker 在*自己的*上下文里跑 `tsc && test`,自修到绿(最多 3 次)才返回。orchestrator 绝不审查未过闸的代码。
4. **只审 diff** —— orchestrator 审的是设计和意图,不是语法和类型——那些归闸门管。
5. **增量重试** —— 返工发回*同一个* worker,只带增量说明,不起新 worker 重发全量任务包。

## Benchmarks(实测数据)

所有数字来自真实的 headless `claude -p` 运行(主循环 = Claude Fable 5,worker = Sonnet 5 / Haiku 4.5),通过 CLI 的分模型用量 JSON 直接测得,零估算。每个运行都通过了全部闸门(`tsc --noEmit` + `vitest`)和全部质量断言——**有没有 skill,产出质量完全一样**,差别只在成本和时间。

### 小任务:skill 反而更亏(这是诚实的结论)

三个小任务(≤ ~150 行改动、≤ 5 个文件):

| 任务 | Fable output token(用skill) | Fable output token(不用) | Δ Fable | 总费用 用/不用 | 耗时 用/不用 |
|---|---|---|---|---|---|
| 多文件功能(hook + 测试) | 4,277 | 2,757 | **+55%** | $1.25 / $0.78 | 121s / 53s |
| 全仓库机械改名 | 1,865 | 1,124 | **+66%** | $0.82 / $0.63 | 83s / 36s |
| API 错误信封设计+全量应用 | 6,415 | 4,798 | **+34%** | $1.85 / $1.08 | 235s / 84s |

在盈亏平衡点以下,"写任务包 + 审 diff"花掉的贵模型 token 比直接把改动敲出来还多。质量断言两组各 30/30 全过——没有任何质量增益来抵消这笔开销。

**委托门槛(已写进 skill 本体):** 只在满足以下任一条件时才委托——预计 ≥ ~300 行新增/改动,或 ≥ ~6 个文件,或跨大量调用点的重复性修改。低于这条线,skill 会拒绝编排,直接在主循环里干。

### 大任务:四种跑法全对照

同一个大体量任务——从零构建 8 模块购物车子系统(约 1,100 行代码 + 测试)——跑了四种配置。四组全部闸门通过、质量断言满分(8/8 模块、测试全绿):

| 配置 | Fable output | Fable 费用 | 总费用 | 耗时 | 测试数 |
|---|---|---|---|---|---|
| 基线——Fable 全干 | 30,993 | $3.05 | $3.05 | 335s | 81 |
| **lite**——Fable 统筹,Sonnet 实现 | 17,899(**−42%**) | $2.03 | **$2.91(最低)** | 465s | **91** |
| **max**——Opus 4.8 主循环 + Fable 顾问 | **3,278(−89%)** | **$0.38** | $5.66 | **116s** | 83 |
| max 变体——Sonnet 5 主循环 | 12,152(−61%) | $1.41 | $5.87(最高) | 162s | 66 |

### Fable 额度账单,逐项列清

上文的"Fable 节省"引用的是 output token,但订阅限额计量的是**全维度加权用量**——输入、输出、缓存写、缓存读都算。下面是同一任务里每种配置记在 Fable 名下的全部账目:

| 配置 | Input | Output | 缓存写 | 缓存读 | Fable 名下费用 | 额度消耗 vs 基线 |
|---|---|---|---|---|---|---|
| 基线 | 11,861 | 30,993 | 49,273 | 398,157 | $3.05 | — |
| lite | 11,595 | 17,899 | 40,451 | 205,528 | $2.03 | **−34%** |
| max(Opus 主循环) | 11,046 | 3,278 | 8,282 | **0** | $0.38 | **−88%** |
| max(Sonnet 主循环) | 22,614 | 12,152 | 41,098 | 65,919 | $1.41 | −54% |

官方没有公开额度加权公式,但单价对各维度的加权和算力成本是同构的——所以 **"Fable 名下费用"这一列是额度消耗的最佳可用代理指标**,本 README 所有"节省百分比"都锚定在它上面。

注意 max-opus 那行缓存读的 **0**:顾问完全不携带会话上下文——两次无状态的"简报进、裁决出",和协议设计分毫不差。9 倍的削减主要来自这里,而不只是 output 变少:在基线和 lite 里,最强模型光是*坐在主循环上*,每一轮都要重读几十万缓存上下文 token。

三条诚实结论:

1. **max 档兑现了承诺——但只对额度。** Opus 主循环下 Fable 只花 3,278 token(两次干净的顾问检查点),消耗降 9 倍;但总费用**反升 86%**:主循环税没消失,只是搬给了 Opus 且变大了(同样的统筹角色,Opus 输出 60k,Fable 只要 17.9k)。
2. **主循环模型的纪律比单价更重要。** Sonnet 5 单价只有 Opus 的六成,却抱着 Fable 顾问不撒手(3.7 倍消耗)、cacheRead 冲到 460 万——额度和美元两头输。max 档的推荐主循环是 **Opus 级**。
3. **这个任务规模下,没有任何编排配置在总费用上赢过基线。** lite 最接近(−5%)。如果美元是你唯一的指标,编排只在远大于 1,100 行的任务上才可能回本——先想清楚你优化的是哪个钱包,再打开这个 skill。

**两张表放在一起看,结论很简单:** 在委托门槛以下,skill 是在给你加税;在门槛以上,它把你最贵档的消耗砍掉三分之一到接近一半,代价是更长的墙钟时间。

### 你省的到底是什么?(如果觉得 −5% 没意思,读这段)

存在两个完全不同的钱包,这个 skill 对其中一个的价值大得多:

- **按 API token 付费的用户** —— 总费用是你的唯一指标,而在盈亏点附近降幅确实有限(−5%):实现工作总得*有人*干,Sonnet 的那份也要钱。费用差距随任务规模拉大——orchestrator 的任务包+审查开销基本固定,下放的实现量却随任务增长。如果你是这类用户,只在真正的大任务上用它。
- **订阅制用户(Claude Max 等)** —— 你的真实约束是最强模型的**限额**,不是美元。便宜档烧额度慢得多,而额度窗口到点就重置,省下的钱不退。**最强档消耗 −42% 意味着:同一个额度窗口能多干约 1.7 倍的编排工作。** 这才是 skill 的主要受众:它首先是**额度套利工具**,其次才是省钱工具。

### 测试方法

- 每组条件都是独立的 headless `claude -p` 会话,跑在同一 fixture 仓库的全新拷贝里(真实 `pnpm` 项目,已 git 初始化)。
- with-skill 的 prompt 要求调用 skill;baseline 的 prompt 禁用一切 skill,任务文本完全相同。
- 费用/token 来自 CLI `--output-format json` 的分模型用量明细,非估算。
- 两组条件用完全相同的客观断言评分(重跑闸门、grep 验证、diff 范围检查)。

## 什么时候别用它

一行改动、纯分析讨论、设计密集型核心、以及低于上述委托门槛的一切任务。skill 自己会说"这个我不该接"然后让位——为什么,看上面的数字。
